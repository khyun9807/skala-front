이체 기능을 만들다가 회계까지 보게 된 날

처음에는 이체 기능을 구현하면 끝이라고 생각했다.
계좌 잔액을 바꾸고, 거래 내역을 남기면 “뱅킹 시스템은 여기서 끝나나?” 정도였다.

그런데 막상 대량 이체 상황을 가정해보니 생각이 달라졌다.
잔액이 맞는 것과, 자금 흐름을 증명할 수 있는 것은 다른 문제였다. 이체가 성공했는데 분개 기록이 누락되거나, 일부만 저장되면 겉으로는 멀쩡해 보여도 내부적으로는 꽤 위험한 상태가 될 수 있었다.

그래서 단순 거래 기록이 아니라 복식부기 원장 구조를 붙이고, 차변과 대변이 맞는지 검증하는 흐름을 만들었다. 이때 느낀 건 금융 시스템에서는 “처리했다”보다 처리 결과를 추적하고 검증할 수 있다가 훨씬 중요하다는 점이었다.

어려웠던 부분은 검증을 어디까지 자동화할지였다.
모든 이상 상황을 사람이 직접 확인하게 하면 늦고, 반대로 시스템이 너무 많은 판단을 하게 하면 운영자가 원인을 파악하기 어려울 수 있었다. 그래서 이상 징후를 감지하면 로그와 알림으로 운영자가 바로 확인할 수 있는 구조를 고민했다.

마침 모니터링을 위해 Grafana를 사용 중이었고, Grafana는 메신저와 연결해 특정 조건이 발생하면 메세지를 보낼 수 있는 기능을 제공했다.

이번 작업을 하면서 금융 도메인의 무게를 조금 체감했다.
API 응답이 빠른 것도 중요하지만, 돈이 오가는 시스템에서는 결국 정합성, 추적 가능성, 운영 대응성이 같이 따라와야 했다.
덕분에 이체 기능을 단순 CRUD가 아니라, 기록과 검증까지 포함한 하나의 처리 체계로 보게 됐다.
