JPA만으로는 부담됐다.

![](https://velog.velcdn.com/images/khyun9807/post/8a99aa00-a422-4194-97aa-19c61141b290/image.jpg)

MyBatis를 적절히 활용해보자!

이체 처리를 구현하면서 처음에는 JPA만으로도 충분하다고 생각했다.
트랜잭션 관리도 편하고, 엔티티 관계를 코드에서 다루기 쉬워서 유지보수 측면에서는 장점이 컸다.

그런데 대량 이체를 가정하고 부하를 걸어보니 병목은 조회보다 쓰기 경로에서 먼저 드러났다.
이체 1건을 처리하는 동안 거래 결과, 원장 전표, 원장 분개, 계좌 잔액 갱신이 이어졌고, 이 과정에서 여러 INSERT와 UPDATE가 반복됐다.

특히 JPA의 IDENTITY 전략에서는 INSERT 직후 DB가 생성한 ID를 받아와야 해서, 여러 엔티티 저장이 개별 INSERT 중심으로 실행되는 한계가 있었다. 그래서 쓰기 작업이 많아질수록 DB 실행 요청이 늘고, 계좌 락을 점유하는 시간도 길어졌다.

처음에는 구조를 크게 바꿔야 하나 고민했지만, 전부 바꾸는 건 오히려 위험하다고 판단했다.
그래서 JPA의 트랜잭션 관리와 기존 도메인 구조는 유지하고, 반복되는 쓰기 구간만 MyBatis bulk 연산으로 분리했다.

이번 작업을 통해 느낀 건, **기술을 하나로 통일하는 것보다 구간별 특성에 맞게 도구를 선택하는 게 더 중요하다는 점**이었다. JPA는 객체 중심 처리에 강점이 있지만, 대량 쓰기처럼 SQL 제어가 중요한 구간에서는 MyBatis가 더 적합할 수 있었다.

결국 이번 개선은 단순히 쿼리를 줄인 작업이 아니라, 편한 구조와 빠른 구조 사이에서 균형을 잡아본 경험이었다. **성능 문제를 볼 때는 어느 구간에서 어떤 비용이 발생하는지 먼저 확인해야겠다고 느꼈다.**
