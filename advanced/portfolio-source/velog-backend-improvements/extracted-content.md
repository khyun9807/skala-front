# Payper — 위치 기반 카드 추천 및 혜택 공유 서비스
(이미지 헤더: "payper-위치 기반 카드 추천 및 혜택 공유 서비스-N")

## 01. 쿼리 최적화 — 단건 응답시간 약 2,000ms → 약 60ms
**제목**: 카드 검색 쿼리의 반복 스캔·불필요 조인·텍스트 검색 비효율을 개선해 단건 응답시간을 약 2,000ms에서 약 60ms로 단축

**상황**
- [검색 조건 해당 카드 ID 10건 조회] 쿼리, [카드 + 부가 정보 가져오기] 쿼리 모두 병목 발생
- [카드 ID 10건 조회] 문제점:
  - 조건절 EXISTS 서브쿼리가 카드 후보 행마다 반복 평가되어 카드 수·연관정보 수에 비례해 비용 증가
  - LONGTEXT 컬럼에 `%키워드%` 형태 LIKE 검색을 사용해 인덱스를 활용하지 못함
  - 검색 조건이 단순한 경우에도 불필요한 테이블 조인이 고정적으로 수행됨
- [카드 + 부가 정보] 문제점: 혜택·가맹점·카테고리 테이블 조인 과정에서 Full Table Scan 발생해 평균 약 2초 소요

**해결 과정**
- 카드 등록·수정보다 검색 요청 비중이 높아, 조회 패턴과 조인 경로에 맞춘 인덱스 재설계
- [카드 ID 10건 조회] 쿼리 수정: 카드별 EXISTS 반복 평가 → 연관 정보를 1회 조인 후 조건 필터링으로 카드 ID 추출하는 구조로 변경
- 검색 절 LONGTEXT 컬럼에 FULL TEXT INDEX 적용
- MyBatis 동적 SQL로 검색 조건에 따라 필요한 조인만 수행
- 카드 ID 조회 결과를 후속 상세 조회의 FROM 절 서브쿼리로 결합해 DB 라운드트립 1회 감소

**결과**
- 반복 스캔 비용 감소(단일 조인 기반 처리로 전환)
- 주요 조회 경로의 Full Table Scan 제거(조인·조건·텍스트 검색 컬럼에 인덱스)
- 슬로우 쿼리 구간이던 단건 조회 응답시간 약 2,000ms → 약 60ms

## 02. Caffeine 인메모리 캐시 — 피크 910ms → 157ms
**제목**: 카드 검색에 Caffeine 인메모리 캐시와 키 최적화를 적용해 DB 커넥션 포화를 완화하고 피크 응답시간을 약 910ms에서 약 157ms로 단축

**상황**
- 한정된 DB 커넥션 수·서버 자원, 조회 트래픽 급증 시 DB 커넥션 풀이 먼저 고갈되어 응답 성능 저하, 인프라 증설만으로 대응하기엔 비용 부담 큼
- 500 req/s 부하 환경에서 평균 응답시간 약 910ms까지 증가
- 카드 등록·수정은 관리자만 수행해 쓰기 비중 낮고, 대부분 트래픽이 조회에 집중
- 검색 조건 조합이 제한적이고 동일 조회 패턴이 반복적으로 발생

**해결 과정**
- DB 접근 감소 위해 Caffeine 기반 인메모리 Look-Aside 캐시 도입
- 초기에는 @Cacheable 키에 카드사·키워드·가맹점·카테고리 등 모든 조건 조합 포함 → 검색 키워드까지 키에 포함하며 조합 수 과도 증가로 히트율 저하·메모리 낭비
  - → 정형 조건 조회만 캐시 대상으로 두고, 키워드 검색은 캐시 바이패스로 분리
- Caffeine TTL·엔트리 크기 조정, 카드 등록·수정 시 @CacheEvict로 캐시 무효화

**결과**
- DB 커넥션 풀 포화 상태 지속 시간 약 75% 감소
- 키워드 검색을 캐시 대상에서 분리해 평균 캐시 히트율 약 97% 확보
- 피크 타임 평균 응답속도 약 910ms → 약 157ms

# Payper Community — 가맹점 기반 혜택 정보 제공 및 공유 커뮤니티
(이미지 헤더: "payper community-가맹점 기반 혜택 정보 제공 및 공유 커뮤니티-N")

## 03. 캐시 스탬피드 완화 — 평균 1.3초 → 217ms
**제목**: 게시글 상세 조회의 캐시 스탬피드를 Jitter·Redis 락·논리적 TTL로 완화해 평균 응답시간을 약 1.3초에서 217ms로 단축

**상황**
- 게시글 상세 조회 DB 커넥션 부족 발생해 Redis 기반 Look-Aside 캐시 도입
- 게시글 조회수 반영 과정에서 Row Lock 대기와 DB 커넥션 부족 발생
- 게시글 상세 조회 캐싱과 조회수의 Redis 기반 Write-Behind 반영으로 평균 응답속도 60% 개선한 상황
- 캐시 만료 시점에 동일 게시글 요청이 몰리며 캐시 미스율 약 86%까지 증가, DB 조회 급증
- DB 커넥션 점유·대기 시간 급증으로 다른 기능까지 커넥션 대기 유발 → 서비스 전반 성능 저하

**해결 과정**
- 게시글 상세 페이지 캐시 TTL에 Jitter(±50%)를 적용해 다수 키의 동시 만료를 분산
- 캐시 만료 시 동일 키에 대해 여러 요청이 동시에 DB 조회·Redis 재적재하는 중복 재생성 발생
  - → 캐시 재생성 시 Redis 분산 락을 적용해 동일 캐시 키 재생성이 1회만 수행되도록 제어
- 물리적 TTL(Redis 실제 보관 시간)과 논리적 TTL(앱이 유효하다고 판단하는 시간)을 나누어, 논리 TTL이 지난 데이터를 stale 상태로 구분 관리
- (캐시 흐름) 물리적 MISS → SETNX 캐시 락 → 락 획득 시 DB조회 후 Redis 저장·반환 / 락 실패 시 대기 후 재시도. 물리 HIT + 논리 만료 전 → FRESH 반환. 물리 HIT + 논리 만료 → SETNX 락 획득 시 DB조회·갱신 후 OLD 반환, 락 실패 시 OLD 반환

**결과**
- 여러 캐시 키 만료 시점 분산으로 서비스 부하 최소화, 다른 서비스 성능 저하 완화
- 캐시 키 만료 후 같은 키 재생성 목적 DB 접근 N회 → 1회 (동일 키 만료 시 중복 DB 접근 약 75% 감소)
- 재생성 락 획득 실패 요청에는 stale 데이터 즉시 반환해 불필요한 락 대기 제거
- 평균 응답속도 약 1.3s → 약 217ms, 약 628 TPS → 약 1,000 TPS

## 04. Circuit Breaker 기반 장애 대응 (좋아요·Top100)
**제목**: 게시글 좋아요·Top100 조회에 Redis 캐시와 Circuit Breaker 기반 장애 대응을 적용해 성능과 서비스 연속성을 개선

**상황**
- 게시글 좋아요 토글과 Top100 랭킹 조회가 모두 요청마다 DB 조회·갱신에 의존 → 트래픽 증가 시 DB 커넥션 풀 고갈·평균 약 500ms 응답 지연 병목
- Redis 기반 Write-Behind(비트맵[게시글-사용자] + 카운터[좋아요 수] + Sorted Set[랭킹])로 대응
- Redis를 핵심 경로로 사용하면서 장애 발생 시 좋아요 토글·랭킹 조회가 모두 실패하는 문제
- DB 반영 스케줄러가 Redis 예외를 처리하지 못해 실행 중단 후 재시작 전까지 복구되지 않는 문제

**해결 과정**
- Redis 의존 경로를 Facade 계층으로 추상화하고 Resilience4j Circuit Breaker 적용
- Circuit Breaker OPEN 시 좋아요 처리·랭킹 조회를 DB 기반 경로로 우회하는 fallback 구조 구현
- 상태 전환 리스너 등록: HALF_OPEN → CLOSED 복귀 시 DB 기반 Top100 데이터를 Redis에 자동 재적재하는 복구 구조
- DB 반영 스케줄러에 TRY-CATCH 추가해 영구중단 방지, Redis 복구 시 자동 동기화
- (상태 다이어그램) CLOSED →(Redis 실패율 50%↑)→ OPEN →(30초, Redis 복구예상)→ HALF OPEN →(시험 성공/DB TO REDIS SEED RUN)→ CLOSED

**결과**
- 좋아요·랭킹 조회 핫패스에서 DB 접근 제거 → DB 부하 완화·응답속도 대폭 향상
- Redis 장애 시 Circuit Breaker가 평균 5회 호출 내 OPEN으로 전환, 무중단 서비스 유지
- Redis 복구 시 30초 내 CLOSED 자동 복귀 및 Top100 데이터 시딩 → 수동 개입 없이 자동 복구

## 05. Outbox·Dispatcher-Worker·동적 폴링 (선착순 이벤트)
**제목**: 선착순 이벤트 후처리에 Outbox·Dispatcher-Worker·동적 폴링을 적용해 응답 지연·장애 전파·작업 경합을 줄인 비동기 처리 구조 개선

**상황**
- 선착순 이벤트 참여·집계·쿠폰 발행이 하나의 트랜잭션에서 동기적으로 처리
- 집계·쿠폰 발행 지연이 사용자 응답시간에 직접 반영되어 후처리 병목이 전체 처리량 상한 결정
- 집계·쿠폰 발행 실패가 전체 트랜잭션 롤백으로 이어져 정상 처리된 이벤트 참여까지 취소(장애 전파)
- 후처리를 단순 병렬화 시 모든 워커 스레드가 Outbox 테이블에 동시 접근해 조회·락 경합
- 폴링 주기마다 일부 스레드가 대부분 작업 선점, 나머지 유휴 → 불균형

**해결 과정**
- 집계·쿠폰 발행을 Outbox 이벤트로 기록해 이벤트 참여 트랜잭션의 동기 경로에서 분리
- @Scheduled 기반 폴링으로 Outbox 이벤트 비동기 처리
- 다중 워커 직접 조회 → 단일 Dispatcher가 Outbox 조회, Worker들이 큐(LinkedBlockingQueue capacity 1000~2000)의 작업을 병렬 처리
- 직전 폴링 처리량에 따라 다음 폴링 주기 조정하는 동적 폴링(50ms~30s)으로 유휴 시 불필요 조회 감소·피크 시 반응성 확보

**결과**
- 이벤트 참여 처리와 후처리 분리로 사용자 응답시간 단축·실패 격리 달성
- Dispatcher-Worker 패턴: 워커 수 증가에 따라 처리량 확장, 이벤트 균등 분배로 처리량 약 2.9배 증가
- 동적 폴링으로 유휴 시 폴링 쿼리 분당 60회 → 최소 2회로 감소

# Bridge Bank — 대규모 이체 트래픽을 가정한 뱅킹 서버
(이미지 헤더: "bridge bank-대규모 이체 트래픽을 가정한 뱅킹 서버-N")

## 07. MyBatis bulk 쓰기 최적화 — 처리량 3.5만 → 5.8만건/분
**제목**: 이체 처리에서 JPA IDENTITY 한계를 MyBatis bulk 연산으로 보완해 DB 라운드트립 50% 감소와 처리량 3.5만건/분→5.8만건/분 개선

**상황**
- 계좌 비관적 락 점유 구간에서 이체 1건당 12회의 DB 라운드트립 발생해 락 점유 시간 증가
- JPA IDENTITY 전략에서 Hibernate가 INSERT 즉시 생성 ID를 필요로 해 saveAll() 호출 시 배치 처리 불가
- SEQUENCE 전략 전환 시 allocation size 관리·테스트 데이터 정합성 유지 부담 커져 변경 비용 증가

**해결 과정**
- 쓰기 경로를 JPA에서 우회해 MyBatis multi-row values insert 적용, INSERT 라운드트립 8회 → 3회
- 계좌 잔액 갱신에 단일 Bulk Update 적용, UPDATE 라운드트립 2회 → 1회
- SEQUENCE 전환 대신 기존 IDENTITY 전략·테스트 코드·설정은 유지하고 쓰기 경로만 교체하는 최소 변경 구조

**결과**
- 이체 1건당 총 라운드트립 50% 감소(INSERT 62.5%↓, UPDATE 50%↓)
- 단건 이체 처리 시간 약 120ms → 약 33ms
- 이체 데이터 정합성 유지하며 분당 이체 처리량 약 3.5만 건 → 약 5.8만 건
- 기존 JPA 영속성·트랜잭션 관리 유지 + MyBatis 쓰기 최적화만 추가한 하이브리드 구조

## 08. Union-Find 파티셔닝 데드락 방지 — 예약이체 처리량 → 1만건/분
**제목**: 예약 이체 처리에 Union-Find 기반 파티셔닝과 Dispatcher-Worker 병렬 구조를 적용해 데드락·락 경합·중복 실행을 방지하고 처리량을 1만건/분으로 개선

**상황**
- 분당 1만 건 처리 목표 대비, 단일 스레드 순차 처리는 약 4,000건/분 수준 처리량 한계
- 계좌 데이터에 비관적 락 적용 상태에서 병렬 처리 시 다중 워커가 동일 계좌를 교차로 잠그며 데드락 가능성
- 해시 기반 파티셔닝은 동일 계좌를 공유하거나 연쇄 연결된 이체 건을 서로 다른 파티션으로 분산시켜 데드락·락 경합 유발
- 이전 스케줄 실행이 끝나기 전에 다음 사이클이 시작되면 동일 이체 건 재조회·재분배되어 중복 실행 위험

**해결 과정**
- Union-Find 알고리즘으로 연관 계좌를 공유하는 이체 건을 동일 파티션으로 그룹화
- Dispatcher-Worker 패턴 적용, 파티션 간에는 CompletableFuture로 병렬 실행하되 파티션 내부는 순차 처리
- 스케줄러가 `CompletableFuture.allOf().join()`으로 현재 사이클 종료를 대기하도록 해 중복 실행 방지
- (파티셔닝 비교) HASH 기반은 데드락/락경합/순서보장X, Union-Find 기반은 데드락X·락경합X·순서보장O

**결과**
- 파티션 간 공유 계좌 제거로 데드락·락 경합 구조적 완화
- 파티션 내부 순차 처리로 연관 이체 간 실행 순서 보장
- 분당 평균 예약 거래 처리량 약 400건 → 약 1만 건
- 약 1만 건/분 부하 테스트에서 이중 이체 0건, 차변·대변 합계 일치 확인

## 06. 복식부기 원장·회계 검증·이상 알림 체계
**제목**: 이체 거래에 복식부기 원장과 회계 검증·이상 알림 체계를 구축해 회계 정합성과 운영 대응성을 강화

**상황**
- 잔액 갱신·거래 내역만 존재해 자금 흐름의 회계 정합성을 증명할 수 없는 구조
- 이체 성공 후 분개 기록이 누락·중복·오류 상태가 되어도 감지할 수 없는 상태
- 거래 유형 각각에 대한 독립적 회계 검증 어려움
- 불일치 발생 시 운영자가 즉시 인지·대응할 수 있는 알림 체계 부재

**해결 과정**
- 고객 예금을 부채, 보유금을 자산으로 보는 은행 회계 원칙에 따라 복식부기 원장 모델(전표·분개) 설계
- 이체 처리와 원장 기록을 단일 트랜잭션으로 묶어 회계 기록 누락 없는 원자적 저장 구조
- 거래 유형별 차변 합계·대변 합계·거래 금액 합계를 자동 교차 검증하고 이상 발생 시 운영자 알림 자동 발송 파이프라인 구축 (모니터링 GRAFANA → 회계 에러로그 감지 → SLACK → 운영자 확인·대응)
- 향후 처리량 확장 위해 회계 기록 기능을 Outbox 패턴으로 분리하는 구조도 검토

**결과**
- 분당 약 11.7만 건 전표, 23.5만 건 분개 처리 환경에서도 차변·대변·거래 금액 합계 전량 일치 검증
- 이체와 원장 기록을 동일 커밋으로 처리해 부분 실패 시나리오 원천 차단
- 자동화된 회계 검증과 이상 발생 시 즉시 운영 대응 체계 확보
