## Troubleshooting 1 — 2,807개 시중 카드 상품 데이터 입력 자동화 (슬라이드 18·19) · [팀원 담당]

**(1) 문제 — 수작업 입력의 한계**
- 정해둔 스키마에 카드 데이터를 직접 입력하려 했으나, 카드 연관 테이블이 **7개**, 카드당 혜택이 **4~10개**씩 얽혀 있어 **카드 한 장 입력에 약 11분** 소요.
- 시중 카드 2,500개 입력 = **총 458시간** → 현실적으로 불가능.
- → **크롤링 도입.** 카드사별 포맷이 모두 달라 직접 접근이 어려워, 카드 정보가 집합된 **카드고릴라(CardGorilla)**를 대상으로 선정. (크롤링 JSON 예: `idx/cid/cate:"CHK"/corp{name:"KB국민카드", name_eng:"kb", color, logo_img{...}}`)

**(2) 해결 — 크롤링 + 정제 + 매칭**
- 카드별 크롤링 데이터를 **Jsoup**으로 정제.
- 기존 등록된 가맹점·카테고리와 매칭하여 저장. **동일 가맹점명 유사 명칭 매칭**(예: 베라, 배라, baskin robbins … → 베스킨라빈스).
- 최종 약 4분간 2,885건 중 **2,813건 크롤링, 2,807건 저장 성공(97% 이상).**
- **카드 1장 입력 기준 11분 → 84.4ms, 7,820배 속도 증가.** (DB card 테이블에 실제 입력됨)

## Troubleshooting 2 — CODEF 성능/비용 문제 파악 및 해결책 설계 (슬라이드 20·21) · [팀원 담당]

**(1) 문제**
- 기존 구현은 **카드 사용 내역을 조회할 때마다 CODEF API를 호출**.
- 외부 API 응답이라 **느린 응답(7.15s)** 으로 사용자 경험 저하, 게다가 호출이 **100번(CODEF 무료 버전)** 으로 제한되는 문제.
- → "CODEF 호출을 줄여보자!"

**(2) 해결 — 결과 캐싱(별도 테이블)**
- 응답 결과를 별도 테이블 **`user_card_transaction`** 에 저장.
- 사용 내역을 DB에서 조회해 **있으면 바로 반환, 없으면 CODEF API 호출**(Cache-Aside).
- **응답 속도 7.15s → 271.18ms, 26.29배(96.20%) 개선**, 100회 호출 제한 문제도 해결.
- (Response Time 계측: Waiting/TTFB가 7.15s → 267.30ms로 대부분의 개선이 여기서 발생)

## ★ Troubleshooting 3 — Docker + Grafana·k6로 원격 서버 유사 부하 테스트 환경 구성 (슬라이드 22) · [내 담당]

- 부하 테스트는 원격 환경 변수를 통제하기 위해 **로컬에서 우선 진행**.
- 로컬 PC와 실제 원격 서버(**EC2 Free Tier**)의 가용 리소스가 달라, **Docker 컨테이너 옵션**으로 최대한 맞춤.
- 원격 서버와 비슷한 **1 CPU, 1GB 메모리(스왑 포함 3GB)** 를 시뮬레이션.
- 실행 예:
  ```
  docker run --memory="1g" --memory-swap="3g" --cpus="1" --cpuset-cpus="0" \
    --net payper-net --name app -d -p 8080:8080 \
    -e JDBC_URL=jdbc:mysql://mysql:3306... payper-server
  ```
- (Docker Desktop Stats로 CPU/Memory/Disk/Network I/O 모니터링)

> 이 부하 테스트 환경이 TS4·TS5 및 velog 개선일지의 성능 측정 기반이 됨. [`../CAREER-SUMMARY.md`](../CAREER-SUMMARY.md) payper 성과 4번(Prometheus·Grafana·k6·Docker 자원 제한 테스트 환경)과 동일.

## ★ Troubleshooting 4 — 시중 카드 검색 시 OutOfMemoryError 해결 + API 성능 개선 (슬라이드 23·24·25) · [내 담당]

**(1) 문제**
- 카드 데이터를 모두 넣고 시중 카드 검색 API 요청 시 **OutOfMemoryError와 함께 서버 다운.**
- 해당 API가 **`SELECT *` SQL로 매번 모든 데이터를 메모리에 로드**하는 구현이 원인.
- 부하 테스트(*12분간 VU 1명→1000명 증가, 초당 1회/VU)로 **P99(하위 99%) 응답 지연** 기록. (Grafana Heap 97.6%)

**(2) 해결**
- **Offset 기반 페이지네이션** 도입으로 OOM 1차 해소.
- 부하 테스트로 다양한 요청 시나리오의 **인덱스 활용률 점검 → 인덱스 튜닝.**
- 빈번하게 쓰이는 검색 조건 기반 **인메모리 캐싱** 도입.
- **결과: P99 응답 속도 기존 7초 → 87ms (감소율 98.8%).**

**(3) GC 지표 개선 (Grafana JVM Statistics)**
- **GC 발생 회수 8회/s → 2회/s**
- **최대 GC(Stop the World) 시간 50ms → 12ms**

> velog 개선일지 01(쿼리 최적화 2,000ms→60ms)·02(Caffeine 캐시 910→157ms)가 이 검색 성능 작업의 세부 기록이며, 모두 내 담당 범위.

## Troubleshooting 5 — 내 카드 조회 API 성능 개선 (슬라이드 26·27·28) · [팀원 담당]

**(1) 문제**
- 핵심 API 중 하나인 **'내 카드 조회'** 부하 테스트(10분간 VU 1→1000명, 초당 1회/VU).
- 평균 응답 **808ms**, P99 **4s**, 처리율 **276.28 TPS**. 테스트 시작 **30초만에 CPU 포화(99%)** 도달.

**(2) 해결 — 단계적 도입**
- 쿼리문 **조인 순서 변경(빠른 필터링)**, 응답 필드 최적화 등 쿼리문 최적화
- **`user_id + deleted`(Soft Delete 여부) 복합 인덱스 추가**
- **`user_id` 기준 인메모리 캐싱**

**최적화 결과 비교 표**
| 지표 | 최적화 전 | 최적화 후 | 개선율 |
|---|---|---|---|
| 처리율(throughput) | 276.28 req/s | 442.49 req/s | 60.16% (1.60배) |
| 평균 응답 시간 | 808ms | 126ms | 84.41% (6.41배) |
| P95 응답 시간 | 2s | 626ms | 68.70% (3.19배) |
| P99 응답 시간 | 4s | 1s | 75.00% (4.00배) |
| 가상 유저(VU) | 175 | 593 | 239% (3.4배) |
| CPU 포화(99% 사용) | 시작 후 30s | 시작 후 8m | 1500% (15배) |

**(3)** CPU 포화 시간 **15배 개선**, 평균 응답 시간 **84.41% 개선** (Docker CPU 사용율·k6 응답 지연 그래프 before/after).

## 포트폴리오 적용 메모

- **내 성과로 쓸 것 = TS3·TS4뿐.** TS4는 "측정(부하테스트·Grafana) → 원인(SELECT * OOM) → 구조 변경(offset 페이징·인덱스 튜닝·캐싱) → 재측정(P99 7s→87ms, GC 50→12ms)" 루프가 완결돼 있어 임팩트가 큼. TS3는 그 측정을 가능케 한 "운영 유사 부하 환경" 근거.
- TS4 + velog 01·02를 묶으면 **payper 검색 성능 개선 스토리**가 완성된다(내 담당).
- TS1·TS2·TS5는 folio에 넣더라도 "팀 성과/맥락"으로만 표기하고, 내 개인 성과 배지로는 쓰지 말 것.
